# אפיון מוצר — פלטפורמת בקרה אישית

**שם עבודה:** "תפריט מבוקר" / מערכת בקרה אישית
**מקרה שימוש ראשון (MVP):** תפריט אכילה מבוקר ע"י הורה
**מסמך עבור:** Claude Code (CODE) — לבנייה
**גרסה:** 1.0 (סגור לבנייה — כל ההחלטות אושרו)

---

## 1. תיאור כללי וחזון

פלטפורמה המאפשרת לאדם אחד (**"המבקר"**) להגדיר, לעקוב ולוודא ביצוע של פעולות שאדם אחר (**"המבוקר"**) אמור לבצע לאורך זמן.

מקרה השימוש הראשון שנבנה הוא **תפריט אכילה** שהורה מגדיר עבור ילד / אדם אחר, אך הליבה תיבנה כך שתתמוך בכל סוג בקרה — נטילת תרופות, שתיית מים, פעילות גופנית ועוד — כולם מבוססים על אותו מנגנון:

> **משימה מתוזמנת → סימון ביצוע ע"י המבוקר → התראה למבקר אם לא בוצע בזמן.**

המטרה: שהמבקר יקבל שקט נפשי וראיה אמינה שהפעולה בוצעה, והמבוקר יקבל מסך פשוט שמוביל אותו צעד‑צעד.

---

## 2. תפקידים (Roles)

| תפקיד | מי זה | מה הוא עושה |
|------|--------|-------------|
| **מבקר** (פותח התפריט) | הורה / מטפל / מלווה | מקים את התוכנית, צופה בדיווחים, מקבל התראות |
| **מבוקר** (האוכל) | הילד / האדם המנוטר | פותח את הפלטפורמה, מסמן ביצוע, מחליף פריט, מצלם |

הערה: אדם אחד יכול להיות גם מבקר וגם מבוקר בהקשרים שונים (למשל הורה שמבוקר ע"י בן/בת זוג בנושא אחר).

---

## 3. רישום והקמת משתמשים

> **ללא OTP.** הכניסה פשוטה — רישום בשם + נייד, וכניסה דרך קישור אישי. אין קוד אימות.

### 3.1 המבקר נרשם
- שם מלא
- מספר נייד

### 3.2 המבקר פותח מבוקר
- שם מלא של המבוקר
- מספר נייד של המבוקר
- המבוקר מקבל הזמנה ב‑SMS עם **קישור כניסה אישי**, ונכנס דרכו ישירות (ללא OTP).

> **אושר:** מבקר אחד יכול לנהל **מספר מבוקרים** כבר ב‑MVP, ומבוקר אחד יכול להיות מקושר למספר מבקרים.

---

## 4. הקמת לוח בקרה (תפריט) — צד המבקר

### 4.1 פרטי הלוח (Board)
בהקמת לוח בקרה המבקר מגדיר:
- **שם ללוח הבקרה** (לדוגמה: "תפריט של דני", "תרופות סבתא").
- **מבוקר** — בחירת המבוקר שאליו משויך הלוח (מתוך רשימת המבוקרים שלו).
- **טווח תאריכים** — תאריך התחלה ותאריך סיום שבהם הלוח פעיל.

### 4.2 שורת תפריט (Plan Item)
לכל שורה מגדירים:
- **יום בשבוע** (א'–ש')
- **שעת אכילה** (שעה מתוזמנת)
- **פריט** (שם המאכל / הפעולה)
- **כמות** (טקסט חופשי או מספר + יחידה, למשל "2 כפות", "100 גרם", "כוס")

ניתן להוסיף כמה שורות לאותו יום (כמה ארוחות / כמה פריטים בארוחה).

### 4.3 שכפול יום
לאחר הקמת **יום אחד** מלא, המבקר יכול ללחוץ **"שכפל לכל השבוע"** — והשורות מועתקות אוטומטית לכל שאר הימים, עם אפשרות לערוך/למחוק נקודתית כל יום אחרי השכפול.

### 4.4 הגדרת סף התראה (X)
ברמת **הלוח** מגדירים ערך **X** — כמה זמן אחרי השעה המתוזמנת, אם השורה לא סומנה כבוצעה, תישלח התראה למבקר (למשל: 30 דקות / שעה).

### 4.5 הגדרת חשיפה למבוקר (Visibility)
ברמת **הלוח** המבקר בוחר כיצד המבוקר רואה את הלוז. שלושה מצבים:
- **יומי** — המבוקר רואה את כל שורות היום.
- **שבועי** — המבוקר רואה את כל שורות השבוע.
- **רק לפי שעה (reveal‑on‑time)** — המבוקר רואה שורה רק כשמגיעה השעה המתוזמנת שלה; לפני כן היא מוסתרת.

### 4.6 הגדרת חובת צילום (Photo)
ברמת **הלוח** המבקר קובע אם צירוף תמונה הוא **חובה** או **רשות** בעת סימון ביצוע. (אם "חובה" — לא ניתן לסמן "בוצע" ללא תמונה.)

---

## 5. ביצוע התפריט — צד המבוקר

### 5.1 מסך המבוקר
התצוגה נקבעת לפי **הגדרת החשיפה** שבחר המבקר (סעיף 4.5):
- במצב **יומי** — המבוקר רואה את כל שורות היום.
- במצב **שבועי** — המבוקר רואה את כל שורות השבוע.
- במצב **רק לפי שעה** — המבוקר רואה שורה רק כשמגיעה שעתה.

כל שורה מציגה: שעה, פריט, כמות, וסטטוס (ממתין / בוצע / הוחלף / פספוס).

### 5.2 פעולות על שורה
לכל שורה המבוקר יכול:
- **לסמן כבוצע** ("אכלתי").
- **להחליף פריט** — לבחור/להזין מה אכל בפועל במקום הפריט המקורי. ההחלפה נשמרת ומוצגת למבקר לצפייה בלבד (המבקר אינו מאשר/דוחה).
- **לצלם** — לצרף תמונה של מה שאכל בפועל. חובה או רשות **לפי הגדרת המבקר** (סעיף 4.6).

### 5.3 עדכון בשעת הארוחה
בהגיע השעה המתוזמנת המבוקר מקבל **תזכורת Push** שהארוחה הגיעה ועליו לסמן.

---

## 6. התראות (Notifications)

| טריגר | למי | מתי |
|--------|-----|------|
| הגעת שעת הארוחה | מבוקר | בשעה המתוזמנת |
| **שורה לא מולאה** | **מבקר** | אם עבר **X** זמן מהשעה המתוזמנת והשורה עדיין לא סומנה |
| ביצוע / החלפה + תמונה | מבקר (אופציונלי) | בזמן אמת כשהמבוקר מסמן |

**ערוץ ההתראות ל‑MVP: Push בלבד.** (SMS משמש רק לקישור ההזמנה הראשוני למבוקר, לא להתראות שוטפות.)

## 7. מודל נתונים (סכימה ראשונית)

```
User
  id, full_name, phone, role_capabilities, created_at

SupervisionLink            // קישור מבקר↔מבוקר (רבים-לרבים)
  id, supervisor_id, monitored_id, status

Plan (לוח בקרה / תפריט)
  id, supervisor_id, monitored_id, name (שם הלוח),
  type ("meal" | "medication" | ...),
  start_date, end_date,                 // טווח תאריכים
  visibility_mode ("daily"|"weekly"|"on_time"),
  photo_required (bool),
  alert_threshold_minutes (X),
  created_at

PlanItem (שורת תפריט)
  id, plan_id, day_of_week, scheduled_time, item_name, quantity

Completion (ביצוע יומי)
  id, plan_item_id, date, status ("pending"|"done"|"replaced"|"missed"),
  replaced_with, photo_url, completed_at

Notification
  id, recipient_id, type, related_completion_id, sent_at, channel ("push")
```

> ההפרדה בין `PlanItem` (התבנית השבועית) ל‑`Completion` (המופע היומי) מאפשרת לשכפל ימים פעם אחת ולייצר אוטומטית את המשימות היומיות, ולשמור היסטוריה.

## 8. מסכים עיקריים

**מבקר:**
1. כניסה דרך קישור אישי (ללא OTP)
2. רשימת מבוקרים (ריבוי מבוקרים)
3. הקמת/עריכת לוח בקרה — שם, מבוקר, טווח תאריכים, שורות, שכפול יום, X, מצב חשיפה, חובת צילום
4. **לוז יומי של המבוקר** — תצוגת כל הבקרות של המבוקר ליום נבחר + סטטוס בזמן אמת
5. לוח מעקב והיסטוריה — ביצועים, החלפות ותמונות
6. מרכז התראות

**מבוקר:**
1. כניסה דרך קישור אישי (ללא OTP)
2. מסך הפעולות — לפי מצב החשיפה שהוגדר (יומי / שבועי / רק לפי שעה)
3. פעולת שורה: סמן / החלף / צלם

---

## 9. הרחבה עתידית — סוגי בקרה נוספים

הליבה גנרית: כל בקרה היא **Plan** מסוג מסוים עם **PlanItems** מתוזמנים ו‑**Completions**. כך נוסיף בקלות:
- **תרופות:** פריט = שם תרופה, כמות = מינון, צילום = תיעוד נטילה.
- **שתיית מים / פעילות / כל הרגל אחר.**

ההבדל בין סוגים יתבטא בעיקר בתוויות (labels), אייקונים ובשדות מותאמים — לא בלוגיקה חדשה.

## 10. הצעת מחסנית טכנולוגית (גמיש — לשיקול CODE)

- **Frontend:** React (Web/PWA), מותאם מובייל, תמיכת RTL ועברית.
- **Backend:** Node.js + REST API.
- **DB:** PostgreSQL.
- **כניסה:** קישור אישי חתום (signed token בקישור), ללא OTP. SMS לשליחת קישור ההזמנה הראשוני בלבד.
- **התראות:** Web Push (Service Worker) — ערוץ ההתראות השוטף.
- **אחסון תמונות:** אחסון אובייקטים (S3 או מקומי ל‑MVP).
- **תזמון התראות:** Job מתוזמן (cron) שבודק שורות שעבר עליהן X זמן ולא בוצעו, ושולח Push.
